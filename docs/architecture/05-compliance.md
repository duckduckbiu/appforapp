## 5.1 为什么要从 Day 1 规划？

```
苹果审核关注的 4 大合规领域:

  1. 内容安全    — 是否有不当内容？有没有举报/屏蔽机制？
  2. 年龄分级    — 是否准确标注？是否有年龄门控？
  3. 用户保护    — 能否举报/投诉？平台是否有审核流程？
  4. 地区法规    — 是否遵守当地法律？（如赌博、加密货币限制）

如果这些功能在 v1.0 之后才加，意味着:
  - 数据库要加新表（影响已有数据）
  - API 要加新端点（影响已有接口）
  - 每个应用的 Manifest 要重新定义（影响已发布应用）
  - UI 到处要改（举报按钮、年龄提示等）

所以：Day 1 把骨架搭好，具体规则和 UI 可以渐进完善。
```

## 5.2 年龄分级系统 (Age Rating)

**Apple Guideline 4.7.5** 要求：宿主 App 必须对每个小程序标注年龄分级。

### 分级标准

| 等级 | 适用内容 | 对应 Apple 等级 | 典型应用 |
|------|---------|----------------|---------|
| `4+` | 无限制内容 | 4+ | 工具、笔记、天气 |
| `9+` | 轻度内容（幻想暴力等） | 9+ | 简单游戏、教育 |
| `12+` | 中度内容（频繁暴力、模拟赌博） | 12+ | 大部分游戏 |
| `17+` | 限制级（真实赌博、成人主题） | 17+ | Rich Game、社交约会 |

### 在 Manifest 中声明

```json
{
  "ageRating": "17+",
  "ageRatingReasons": ["simulated-gambling", "real-money-transactions"],
  "ageGate": true
}
```

### 架构影响

```
平台核心层需要:
  ├── AgeGate 组件 — 用户首次打开 17+ 应用时弹出年龄确认
  ├── 用户生日/年龄字段 — profiles 表加 date_of_birth 列
  ├── 应用商店过滤 — 未成年用户看不到 17+ 应用
  └── Manifest 验证 — 上架审核时检查 ageRating 字段
```

## 5.3 举报与投诉系统 (Report System)

**Apple Guideline 1.2** 要求：用户生成内容（UGC）的 App 必须提供举报和屏蔽机制。

### 举报类型

| 举报对象 | 触发场景 | 举报入口 |
|---------|---------|---------|
| **应用** | 应用包含不当内容、欺诈 | 应用商店详情页 + 应用内 |
| **用户** | 骚扰、仇恨言论、诈骗 | 用户资料页 |
| **内容** | 帖子/评论/消息含不当内容 | 长按内容 → 举报 |
| **频道** | 频道传播违规内容 | 频道详情页 |

### 举报理由枚举

```typescript
type ReportReason =
  | 'spam'                  // 垃圾信息
  | 'harassment'            // 骚扰
  | 'hate_speech'           // 仇恨言论
  | 'violence'              // 暴力内容
  | 'sexual_content'        // 色情内容
  | 'fraud'                 // 欺诈
  | 'impersonation'         // 冒充他人
  | 'underage'              // 未成年用户
  | 'illegal_activity'      // 违法行为
  | 'copyright'             // 侵权
  | 'other';                // 其他
```

### 举报流程

```
用户点击「举报」
  → 选择举报理由（必选）
  → 填写详细说明（可选）
  → 上传截图证据（可选）
  → 提交
  → 平台自动分配审核队列
  → 审核员处理（24h SLA）
  → 结果通知举报人 + 被举报方
```

## 5.4 内容审核框架 (Content Moderation)

### 三层审核机制

```
┌─ 第一层：自动过滤（实时）──────────────────────────┐
│  ├── 关键词过滤（脏话、违禁词、敏感词库）           │
│  ├── AI 文本检测（Perspective API / OpenAI Moderation）│
│  ├── 图片 NSFW 检测（上传时扫描）                   │
│  └── 垃圾信息检测（频率限制 + 模式识别）            │
├─ 第二层：社区举报（异步）──────────────────────────┤
│  ├── 用户举报 → 进入审核队列                       │
│  ├── 累计举报阈值 → 自动隐藏内容（待审核）          │
│  └── 信誉系统 → 高信誉用户的举报权重更高            │
├─ 第三层：人工审核（最终裁决）──────────────────────┤
│  ├── 审核员 Dashboard（Admin 面板新增模块）          │
│  ├── 审核操作：通过 / 删除 / 警告 / 封禁            │
│  └── 申诉流程 → 被处理用户可以申诉                  │
└──────────────────────────────────────────────────┘
```

### 处罚阶梯

| 级别 | 触发条件 | 处罚 | 时长 |
|------|---------|------|------|
| 警告 | 首次轻微违规 | 发送警告通知 | — |
| 禁言 | 2 次违规 或 中度违规 | 禁止发送消息/帖子 | 24h-7d |
| 应用封禁 | 在特定应用内严重违规 | 禁止使用该应用 | 7d-永久 |
| 平台封禁 | 3 次违规 或 严重违规 | 封禁账号 | 30d-永久 |

## 5.5 地区限制框架 (Region Restrictions)

### 限制维度

```
平台级限制（影响整个 Bill.ai）:
  ├── 某些地区完全不可用（如朝鲜、制裁国家）
  └── 某些地区禁用加密货币充值

应用级限制（由应用 Manifest 声明）:
  ├── Rich Game: 在中国大陆、美国部分州不可用
  ├── 理由: 当地赌博法规
  └── 用户看到提示: 「此应用在您所在地区暂不可用」

内容级限制:
  └── 某些内容在特定地区不展示
```

### 架构影响

```
平台核心层需要:
  ├── IP 地理定位服务 — 判断用户所在地区（MaxMind GeoIP 或 Cloudflare）
  ├── 地区字段 — profiles 表加 region / country_code 列
  ├── 应用过滤 — 应用商店列表根据用户地区过滤
  ├── 访问控制中间件 — iframe 加载前检查地区限制
  └── Admin 地区管理页 — 管理平台级和应用级地区规则
```

## 5.6 用户封禁系统 (User Banning)

| 级别 | 范围 | 执行方 | 效果 |
|------|------|--------|------|
| **应用级封禁** | 只影响某个应用 | 应用开发者 / 平台审核 | 无法使用该应用，其他正常 |
| **社交封禁** | 禁止社交功能 | 平台审核 | 不能发帖/评论/聊天，但能用应用 |
| **平台封禁** | 整个 Bill.ai | 平台管理员 | 完全无法登录 |

## 5.7 应用审核流程 (App Review)

```
开发者提交应用
  → 自动检查（Manifest 验证、安全扫描）
  │   ├── Manifest 格式是否正确？
  │   ├── 权限声明是否合理？
  │   ├── 年龄分级是否准确？
  │   ├── 地区限制是否合规？
  │   └── 入口 URL 是否可访问？
  → 人工审核（平台管理员）
  │   ├── 应用功能是否正常？
  │   ├── 内容是否合规？
  │   ├── 是否有安全风险？
  │   └── 用户体验是否达标？
  → 审核结果
      ├── 通过 → 发布到应用商店
      ├── 驳回 → 附带修改意见
      └── 永久拒绝 → 违反不可接受的规则
```

### 应用状态流转

```
draft → submitted → in_review → published
                  ↘ rejected → (修改后) → submitted
published → suspended（违规下架）→ (整改后) → in_review
```

## 5.8 ToS / 隐私政策 / GDPR

以下内容 **不需要从架构层面预留**，纯文本 + 前端页面，随时可加：

| 内容 | 何时需要 | 说明 |
|------|---------|------|
| 服务条款 (ToS) | 上架前 | 纯 Markdown 页面，`/terms` |
| 隐私政策 | 上架前 | 纯 Markdown 页面，`/privacy` |
| Cookie 策略 | 上 EU | 前端弹窗组件 |
| GDPR 数据导出 | 上 EU | 后端 API：导出用户所有数据 |
| GDPR 数据删除 | 上 EU | 后端 API：删除用户账号和数据 |

> 虽然内容可以后写，但 profiles 表的 `accepted_tos_at` 和 `accepted_privacy_at` 字段应该提前预留。
